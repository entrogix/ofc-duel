# ランダムマッチング実現プラン

作成日: 2026-06-14

## 結論

技術的には今のサーバーに**マッチングキューを足すだけ**（半日仕事）。
本当の難関は技術ではなく**「同時接続プレイヤーの確保」**。そのため CPU補充（ボットフォールバック）とセットで設計するのが現実解。

## 実装状況（2026-06-14）

- ✅ **Step 2（マッチングキュー）実装済み** … `join_random` / `cancel_random`、最大4人 or 15秒待ちで自動マッチ
- ✅ **Step 3（過疎対策）実装済み** … CPU補充（人が来なければボットと開始）／配置制限時間60秒＋超過時CPU代行／切断45秒猶予＋トークン再接続／精算後8秒で自動進行
- ⬜ **Step 1（サーバー公開デプロイ）** … あなたの作業。[online-deployment.md](online-deployment.md) 参照
- ⬜ **Step 4（レーティング・戦績）** … 人が付いてから

サーバーのテストは `packages/server` で `npx tsx src/matchmaking_test.ts`（ボット補充・制限時間・再接続の4シナリオ）。

## 全体像

```
スマホ ── wss:// ──> 公開サーバー（Render等にデプロイ）
                      ├─ ルームコード対戦（実装済み）
                      └─ マッチングキュー（追加分）
                          「ランダムマッチ」を押した人を待ち行列に入れ、
                          人数が揃ったら自動でルームを作って開始
```

## 段階的ロードマップ

### Step 1: サーバーを公開する（前提）
[online-deployment.md](online-deployment.md) の通り。Render無料枠でOK（本格運用は$7/月）。

### Step 2: マッチングキュー ✅実装済み
- `join_random` → 待ち行列／`cancel_random` でキャンセル
- 「**最大4人 or 15秒待ち**」で自動開始
- クライアント: オンライン画面に「🌍 ランダムマッチ」ボタン＋待機画面（待ち人数表示）
- アカウント不要・匿名方式のまま

### Step 3: 過疎対策 ✅実装済み
- **CPU補充**: 15秒待って人が揃わなければ「BOT」を混ぜて開始（名前に "BOT" と表示して正直に）
- **放置（AFK）対策**: 配置制限時間60秒（`⏱` バッジ表示）→ 超過で `chooseCpuPlacement` がCPU代行
- **切断対応**: 45秒の再接続猶予＋トークン再接続。猶予中はCPUが代行して進行、猶予切れでボット化
- **自動進行**: ランダムマッチはホスト不在なので、精算8秒後に自動で次ハンド（「次のハンドへ」で前倒し可）
- タイミングは環境変数（`OFC_MATCH_FILL_MS` 等）で調整可能

### Step 4: 育てる段階（人が付いてから）
- 匿名ID（端末ごとのUUID）で戦績保存 → レーティング（Elo）導入 → レート近い人を優先マッチ
- 永続化が必要になる（Render の無料Postgres / SQLite で十分）
- 通報・ブロック、名前のNGワード処理
- 将来的にアカウント連携（Game Center / Google Play Games）

## 正直な課題: プレイヤー人口

ランダムマッチは「押した瞬間に相手がいる」体験が命。
**同時接続が常時10人未満だと、ほぼ確実に誰ともマッチしない。**

個人開発アプリの定石:
1. まず CPU対戦 + 友達対戦（ルームコード）で出す ← 今ここまで完成
2. ランダムマッチは「CPU補充あり」で実装（待たせない）
3. 人が付いてきたら本物同士のマッチ率が自然に上がる

## 技術面の補足

- OFCはターン制なので**遅延はほぼ無関係** → サーバーは東京1リージョンで全世界カバー可能
- Node + ws は1インスタンスで数千接続を捌けるので、当面スケールの心配は不要
- 月コスト目安: サーバー$7 + DB無料枠 = **約1,100円/月**（無料枠なら0円、ただしスリープあり）
